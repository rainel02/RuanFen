3. 项目开发过程总结

概述：本项目遵循需求调研、设计、开发、测试与部署的迭代流程，采用 API-first 思路进行前后端协作。前端基于 Vue 3、Vite、Element Plus，后端以 OpenAPI 规范定义接口，数据库使用关系型存储并在部分场景使用 Elasticsearch 提升检索体验。

需求调研阶段：
分工：产品负责人负责与需求方沟通，技术组参与可行性评估并形成需求文档（OpenAPI）。
收获：明确学者信息管理、成果认领、社交（关注/粉丝）、论坛等核心功能，确定主要技术栈并输出测试数据。
问题：对搜索与索引边界条件理解不足，导致对 Elasticsearch 映射与查询进行了多次优化；早期对审计与严格 RBAC 等安全细节未充分定义。

设计阶段：
分工：架构由资深开发制定，模块接口由后端与前端共同定义，数据库设计由后端完成并产出 ERD 与建表脚本。
收获：形成清晰模块边界（Auth、User/Scholar、Achievement、Forum、Social、Search），并预留审计与权限扩展点；生成 `psq/api.yaml` 的 OpenAPI 文档，便于并行开发与联调。
问题：部分接口最初未充分考虑分页与性能（如粉丝/关注列表），后续补充分页支持并优化查询；缺少统一的安全验收清单（密码哈希、审计策略、CSP/HSTS）。

实现阶段：
分工：多人并行开发后端各模块与前端页面，采用 JWT + Spring Security 进行鉴权，使用 JPA 管理关系型数据，部分场景使用 Elasticsearch 提升搜索体验。
收获：实现用户注册/登录、学者档案、学术成果管理、关注/粉丝管理、论坛发帖回复、全文检索等功能；实现基础接口权限保护与单点登录支持。
问题：密码策略初期采用 BCrypt，未迁移到 Argon2；对敏感操作的审计日志尚未统一实现；XSS 与输入清洗需加强前后端协同。

测试与联调阶段：
分工：测试人员与开发人员协同进行用例覆盖与接口联调，修复逻辑漏洞与并发边界问题。
收获：关键路径（注册、登录、成果认领、关注/取消关注、搜索）完成基本的手工或自动化验证；发现并修复若干权限与异常处理问题。
问题：自动化测试覆盖尚不全面，集成测试与压力测试需加强以提高上线可靠性。

部署与运维阶段：
分工：部署负责人编写部署说明与容器/反向代理示例，生产环境通常由反向代理或平台负责 TLS 终端。
收获：前端可通过 `VITE_API_BASE_URL` 切换环境，应用采用无状态 JWT 便于水平扩展，开发/测试环境可顺利运行。
问题：应用层未在代码中强制 TLS（由运维在部署层完成），生产部署文档需明确 TLS 强制、日志保留与审计策略。

小结：项目在前端交付与接口规范方面进展良好，但为满足生产级安全与合规要求，需补强后端 RBAC、审计记录、密码存储策略以及部署安全配置。

4. 团队总结

团队结构与分工：
产品/需求负责人、架构与后端开发、前端开发、测试、运维（部署）分工明确。关键接口由后端定义，前端与后端通过 OpenAPI 文档对齐与并行开发。

团队优势：
技术栈选择合理，工程化水平高（Vue3、Vite、Pinia、Element Plus），模块化与文档化（JPA/Repository、Swagger/OpenAPI）有助于协作与维护。采用 JWT + Spring Security 快速建立统一鉴权体系，交付速度快，利于早期演示与验收。

存在的问题与改进建议：
安全与合规：需补充审计日志机制（对个人敏感信息修改与资源下载记录操作人/时间/IP/内容），并评估将密码哈希升级为 Argon2 的可行性与迁移策略。
运维与部署：在部署文档中明确 TLS 强制、日志集中化（ELK/云日志）、备份与监控指标。
测试质量：增加单元、集成与压力测试覆盖，尤其针对权限边界、并发与搜索相关场景。
前后端协同：就 XSS 与输出编码达成统一策略（后端输入校验 + 前端输出编码 + CSP）；对用户上传附件/文件进行严格校验与权限控制。

团队文化与流程建议：
定期开展安全审计与代码走查，将权限、异常处理与输入校验作为 PR 审核重点；设立轻量部署/回滚演练流程；将审计与合规需求纳入开发的 Definition of Done（DoD）。
