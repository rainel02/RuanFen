2.确定应用模型
本节将详细阐述“学术成果分享平台”采用的软件架构模式及应用类型，并明确各架构组件与技术选型在系统中的具体部署与实现方式。合理的架构与应用模型选择是保障系统可扩展性、可维护性及性能的关键。
2.1 软件架构模式
本平台采用经典的分层架构模式。该架构通过将系统划分为多个具有明确职责的层次，实现了关注点分离，从而有效地保障了系统的可维护性、可测试性，并降低了层与层之间的耦合度。
2.1.1 分层架构（Layered Architecture）
分层架构将系统功能组织成不同的层次，每一层职责单一且仅与相邻层交互。本平台采用典型的三层架构，包括表示层、业务逻辑层与数据持久层。	
●表示层：作为用户交互的界面，采用 Vue.js 构建响应式单页应用（SPA），并通过基于 Promise 的 HTTP 客户端（如 Axios）与后端的业务逻辑层进行 RESTful API 通信。
●业务逻辑层：作为系统的核心，基于 Spring Boot 框架实现，负责处理所有核心业务规则与流程，例如用户身份认证、学术成果的提交、审核与查询管理等。
●数据持久层：负责系统数据的持久化存储与访问，借助 Hibernate 作为 ORM 框架，对关系型数据库 MySQL 进行封装与管理，实现了对象与关系数据的映射，简化了数据库操作。
典型交互流程：用户在前端表示层提交新的学术成果信息后，请求被发送至后端的业务逻辑层；业务逻辑层对数据进行有效性校验与业务处理；最终，处理后的数据通过数据持久层被持久化存储至 MySQL 数据库中。
2.2 确定应用类型
基于目标用户群体及核心功能需求，“学术成果分享平台”被定义为 Web 应用程序。该类型允许用户通过标准 Web 浏览器进行跨设备、跨平台访问，无需本地安装，具有部署便捷、访问门槛低的优势。平台前端采用 Vue.js 框架构建为单页应用程序，通过动态内容加载与前端路由机制，有效避免了整页刷新，从而提供了接近原生应用的流畅用户体验。
 
3.确定部署策略
在设计系统架构时，我们需要充分考虑服务器的资源限制，确保系统能够在有限的计算资源和内存条件下高效运行。为此，我们对资源密集型操作进行了优化，避免过多的并发请求占用资源。设计上，我们采用单体服务架构，将所有的服务和依赖集中部署到一个服务器集群中，这种方式不仅简化了服务间的通信，也有效减少了资源开销。同时，我们充分考虑系统可能会面临的负载均衡和高并发场景，通过合理的Redis缓存策略优化和k8s的扩缩容机制等方案来进一步提升系统的吞吐能力和稳定性。
3.1部署环境
在选择部署环境时，我们决定使用公有云方案，具体选择了华为云作为部署平台。华为云是国内领先的云服务提供商之一，具有强大的云服务支持、全球化的部署能力和较高的安全性，非常适合应对我们的系统的高并发、高扩展需求。
公有云是指第三方提供商通过公共互联网为用户提供的云服务，用户可以通过互联网轻松访问云平台并享受计算、存储、网络等各类云服务。在公有云环境中，用户只需负责应用和数据的管理，而无需担心硬件管理和运维工作，基础设施由云服务提供商负责，极大地降低了运维成本。
3.2部署方案
考虑到本系统用户规模较小，且经费有限，我们决定使用单体服务架构。单体服务将所有模块放在一个服务器集群中，天然避免了跨服务通信的复杂性。学术成果平台包含成果展示、用户管理、成果搜索等模块，单体架构下这些模块可以直接调用，不需要额外的通信协议，从而降低了延迟，提升性能。此外，单体架构的系统便于进行端到端的整体测试，确保所有功能模块和流程可以协同工作，系统运行稳定。

3.4部署拓扑
鉴于用户使用规模有限，我们选择使用单个集群进行单区域部署，数据中心位于公有云平台。虽然此方案无法有效分担流量，但在用户量较小的情况下，单服务集群足以满足基本的访问需求。所有流量将集中进入该集群进行处理，简化了部署和维护工作。
3.5容器化和编排
在项目中，我们使用 Kubernetes (K8s) 来实现容器化服务的管理和编排。Kubernetes 是一个强大的开源容器编排平台，它能帮助我们在多个环境中快速、可靠地部署应用。通过 K8s，我们可以确保开发、测试和生产环境的高度一致性，简化运维流程，提高部署效率。
在 Kubernetes 中，应用被打包成容器镜像，部署为一个或多个 Pod 进行管理。对于单体架构而言，所有服务和功能将被打包到同一个容器内，K8s 通过管理这些容器的生命周期，确保应用的稳定运行。Kubernetes 的扩缩容机制可以根据负载的变化，动态增加或减少 Pod 数量，以确保系统能够在负载变化时高效运行。
3.6高可用性和容错性
在构建共享学术成果平台时，确保系统数据的高可用性和容错性是至关重要的，因此我们使用 Redis 的主从数据库来保证数据的一致性和抗风险能力。我们设计一个主服务器和一个从服务器之间进行数据同步。主服务器负责处理所有的写入请求，而从服务器则从主服务器复制数据并处理读取请求。
为了保持数据一致性，我们使用异步复制策略。主服务器在处理写请求时，会将数据变更异步地发送给从服务器，但是同时这意味着从服务器在接收到数据更新前，可能会导致短暂的数据不一致，但能显著提高性能。
考虑到Redis属于内存数据库，需要数据持久化，否则在遇到断电系统崩溃时，丢失存储在内存中的所有数据。为此我们采用增量化持久化（AOF）策略和全量式持久化（RDB）策略综合使用的方法。AOF会记录所有的写操作，以追加的方式存储到一个文件中，确保在Redis重启时可以通过重放写操作来恢复数据。我们将AOF的同步策略设置为每秒写入一次，以在性能和数据安全之间取得平衡。虽然这种方式可能导致最近一秒内的数据丢失，但它大大地降低了数据丢失的风险，提供了更好的数据恢复能力。RDB会定期保存全量快照，作为系统故障时的备份。我们配置了RDB的检查点策略，以便定期保存内存中的数据状态，使得系统发生严重故障时，快速恢复到最近的快照状态。虽然RDB恢复速度较快，但由于它是全量备份，不能完全替代AOF的增量备份，因此两者结合使用，确保了系统数据的高可用性和容错能力。

4.2 重要技术说明
4.2.1 Spring Boot
Spring Boot 是基于 Spring 框架的快速开发框架，旨在简化 Spring 应用的配置和部署，通过自动配置和内嵌服务器，减少了繁琐的 XML 配置，使开发者能够专注于业务逻辑，快速创建独立的、生产级的 Spring 应用。Spring Boot 还提供丰富的内置组件和工具，支持微服务架构，适合现代化应用的开发。
 
4.2.2 MySQL
MySQL 是一个开源的关系型数据库管理系统，以其稳定性、易用性和高性能广泛应用于各类应用程序中。它支持结构化查询语言（SQL），适合处理结构化数据，尤其在事务处理、数据一致性方面表现出色。MySQL 提供强大的数据查询和管理功能，支持主从复制、分区等特性，能够在保持可靠性的同时处理大规模数据。
作为一款主流的高可靠性数据库，MySQL可以满足本项目的存储需求，尽管在搜索性能上存在一定局限，但可通过 Elasticsearch 补充其搜索功能。
 
4.2.3 Elasticsearch
Elasticsearch 是一个分布式搜索引擎，基于 Apache Lucene，专为大规模数据处理和高效查询而设计。与 MySQL 等传统关系型数据库相比，Elasticsearch 在全文检索、数据分析和处理速度上有显著优势，尤其适用于高频数据查询和实时分析。它支持自动分片和分布式存储，能够横向扩展，应对大规模数据需求。且通过 RESTful API 接口进行操作，易于与其他系统集成。
本项目使用 Elasticsearch 作为搜索引擎，优化科研成果和人员搜索的效率。
 
4.2.4 Nginx
Nginx 是一个高性能的开源 Web 服务器和反向代理服务器，广泛用于处理高并发的网络请求。它以事件驱动的架构著称，能够高效地管理资源，在处理静态内容、负载均衡和反向代理方面表现出色。相比传统的服务器（如 Apache），Nginx 占用内存更少，性能更高，适合用作流量较大的网站的服务器、缓存服务器和 API 网关。
考虑到本项目的高并发量级和租用服务器的硬件限制，Nginx将是我们的最好选择。
 
4.2.5 Redis
Redis是一个key-value存储系统。它支持存储的value类型相对更多，包括string、list、set、zset和hash。这些数据类型都支持push/pop、add/remove及取交集并集和差集及更丰富的操作，而且这些操作都是原子性的。在此基础上，Redis支持各种不同方式的排序。为了保证效率，数据都是缓存在内存中，Redis会周期性的把更新的数据写入磁盘或者把修改操作写入追加的记录文件。本系统使用 Redis 作为缓存解决方案，以提高系统响应速度并减少数据库负载。

4.2.6 Vue.js
Vue.js是一套用于构建用户界面的渐进式JavaScript框架，其核心库关注视图层，采用自底向上增量开发的设计。Vue 的目标是通过尽可能简单的API实现响应的数据绑定和组合的视图组件。
使用Vue.js作为本项目的前端应用框架，可以更好地支持实时响应式设计。同时，Vue丰富的组件和插件生态可以大幅简化开发的复杂程度。除此之外，利用Vue可以将元素集合封装成组件，方便重复使用和协作开发。
 
4.2.7 Pinia
Pinia 是 Vue 生态系统中的一个轻量级状态管理库，被设计为 Vuex 的替代方案，具有更简单的 API 和更好的 TypeScript 支持。相比 Vuex，Pinia 的配置更简洁，去除了复杂的模块化设置，支持模块热重载和组合式 API，使得状态管理代码更直观、可维护性更高。它在性能和开发体验上都进行了优化，尤其适合现代 Vue 3 项目。
 
4.2.8 Vue-Router
Vue-Router 是 Vue.js 的官方路由管理库，专为单页应用（SPA）设计，支持页面间的导航功能。它支持路由参数、嵌套路由、懒加载等特性，使得开发者可以轻松定义组件与 URL 的映射关系，实现无刷新页面切换和动态路由加载，是构建 Vue 应用路由系统的核心工具，帮助实现动态路由加载，优化用户体验。
	
4.2.9 Axios
Axios 是一个基于 Promise 的 HTTP 客户端，用于发送 HTTP 请求，广泛应用于前端与后端的通信。它支持多种 HTTP 请求方式（如 GET、POST）以及请求拦截、响应拦截等功能，简化了网络请求的处理流程。本项目在前端部分使用 Axios 来与后端服务进行 API 通信，它的异步特性使得处理 API 请求变得更加直观和高效，同时有助于管理请求的错误和响应数据。

5.1.1概念完整性
 
使用统一的命名规范：为保证代码的可读性和团队协作效率，项目遵循统一的命名规范。前端Vue变量与方法采用camelCase驼峰式命名，组件文件采用PascalCase命名；后端Spring Boot的类采用PascalCase命名，方法和变量采用camelCase命名；MySQL数据库表和字段采用snake_case命名；前后端数据交互的JSON字段统一采用camelCase命名。
使用统一的术语：项目中涉及的核心概念，如“文献(Document)”、“科研人员(Researcher)”等，在前后端代码、数据库设计和API文档中保持术语一致，避免产生歧义。
组件化与模块化设计：
前端：项目采用Vue3框架，通过组件化思想构建UI。页面被拆分为可复用的UI组件（如搜索框、文献列表项等），利用Pinia进行集中的状态管理，实现了视图层、状态层和业务逻辑的有效分离。
后端：项目基于Spring Boot框架，采用分层架构设计。严格划分应用层（Controller）、业务逻辑层（Service）和数据访问层（Repository/DAO），实现了“高内聚、低耦合”的设计原则，确保各层职责单一，便于开发和维护。
 
5.1.2可维护性
 
清晰的注释和文档：后端的公开API接口、Service层核心业务方法、复杂的业务逻辑等，均有详细的JavaDoc注释，说明其功能、参数和返回值。前端对关键组件的Props和自定义事件进行清晰的注释。
遵循代码规范：项目团队遵循统一的Java和JavaScript/Vue编码规范，并利用ESLint、Prettier等工具在代码提交前自动格式化和检查，保证代码风格的一致性。
日志记录：后端在关键业务节点和异常捕获处记录详细日志，便于快速定位和解决线上问题。
 
5.1.3可复用性
 
通用组件封装：前端将应用中通用的UI元素（如分页器、模态框、消息提示等）封装为全局Vue组件，可在不同页面中直接复用，提高开发效率。
服务化和工具类：后端将可复用的业务逻辑（如权限校验、文件处理等）抽象为独立的服务（Service）或工具类（Utils），供不同的业务模块调用。
标准化的接口：系统通过RESTful API提供前后端交互接口，定义了清晰的接口规范（URL、HTTP方法、请求/响应格式），使得后端服务不仅能服务于本项目Web端，也能便捷地支持未来的移动端或其他第三方应用。
 
5.2运行时质量
 
5.2.1可用性
 
负载均衡：架构中使用Nginx作为反向代理和负载均衡器。通过部署多个Spring Boot应用实例，Nginx可将请求分发至不同实例，避免单点故障，实现服务的高可用。
容器化部署：整个应用栈（Nginx, Spring Boot, Redis, MySQL, ES）均在Docker容器中运行。当某个服务实例出现故障时，Docker可以快速重启或重新部署新的容器实例，缩短服务中断时间。
数据备份：制定定期的数据库（MySQL）备份策略，确保在发生数据损坏或丢失等灾难性事件时，能够快速恢复数据。
 
5.2.2互操作性
 
采用标准协议：系统对外通过标准的HTTP/HTTPS协议提供服务，接口遵循RESTful设计风格，数据交换格式采用通用的JSON格式，便于与异构系统进行集成和交互。
前后端分离：彻底的前后端分离架构，通过API进行通信，降低了前后端技术栈的耦合度，双方可以独立开发、测试和部署。
 
5.2.3可管理性
 
集中式日志：系统日志（包括Nginx访问日志、Spring Boot应用日志）可被收集到统一的日志管理平台，便于运维人员监控系统运行状态、进行故障排查和性能分析。
 
容器化管理：基于Docker的部署方式简化了应用的部署、升级和环境迁移过程。通过Docker Compose或Kubernetes等编排工具，可以实现对整个应用集群的自动化管理。
 
5.2.4性能
 
缓存加速：引入Redis作为高速缓存，对热点数据（如用户信息、频繁查询的文献元数据等）进行缓存，大幅减少对MySQL数据库的直接访问，显著提升数据读取速度和系统响应能力。
专业搜索引擎：针对文献检索功能，采用Elasticsearch作为搜索引擎。利用其强大的倒排索引和分词技术，实现对海量文献数据的毫秒级全文检索，性能远超传统数据库的模糊查询。
静态资源优化：Nginx除了作为反向代理，还负责高效地处理前端静态资源（HTML/CSS/JS/图片）的请求，减轻了后端Spring Boot应用的压力。
 
5.2.5可靠性
技术栈成熟稳定：选用Spring Boot、Vue3、MySQL、Redis、Nginx等经过大规模生产环境验证的主流开源技术，其社区活跃，稳定性和可靠性高。
 
事务管理：在后端业务逻辑层，对涉及多步数据库操作的业务（如用户注册、文献信息修改等）采用数据库事务管理，保证操作的原子性，确保数据一致性。
代码审查与测试：严格执行代码审查（Code Review）制度，并建立完善的单元测试和集成测试流程，确保代码质量，提前发现和修复潜在的缺陷。
 
5.2.6可伸缩性
 
无状态应用设计：后端Spring Boot应用设计为无状态服务，用户的会话信息等状态通过外部存储（如Redis或JWT）管理，使得应用层可以轻松地进行水平扩展（增加服务器实例）。
水平扩展能力：通过Nginx的负载均衡机制，可以根据业务流量的增长，动态地增加后端的Spring Boot应用容器数量，从而线性提升系统的整体处理能力。
分布式数据存储：Redis和Elasticsearch本身支持集群部署模式，当数据量和并发请求增长时，可以通过增加节点来扩展其存储能力和处理性能。
 
5.2.7安全性
网关层防护：架构图中明确了在网关层进行DDoS防御和SQL注入防御。通过配置Nginx或引入专业的Web应用防火墙（WAF），过滤恶意请求，保护后端服务。
 
认证与授权：系统在网关层（或应用内）实现统一的用户认证（Authentication）和权限鉴定（Authorization），确保只有合法用户才能访问其权限范围内的资源。
 
数据传输加密：系统支持HTTPS协议，对客户端与服务器之间的所有通信数据进行TLS/SSL加密，防止数据在传输过程中被窃听或篡改。
 
敏感信息处理：用户密码等敏感信息在存储到数据库前，必须经过加盐哈希（如BCrypt）处理，不存储明文密码。

6.横切关注点
6.1安全性
6.1.1安全问题
系统在运行过程中可能面临多种安全威胁，包括未经授权的访问、恶意数据注入、信息泄露以及分布式拒绝服务（DDoS）攻击等。若缺乏有效防护，系统的可靠性与用户数据的安全性将受到严重影响。
 因此，本平台重点关注以下安全方向：用户身份认证、访问授权、输入与数据验证、以及系统级安全防护机制，以确保系统在多层面具备稳健的防御能力。

6.1.1.1身份认证
系统采用 基于角色的身份认证机制，平台用户分为两类：
学者：通过注册及邮箱验证后，可登录系统并访问学术资源、文献等功能模块；
管理员：负责系统管理、权限审批、日志审计与安全监控等高级操作。
认证机制采用 JWT 技术实现。
用户登录后，系统根据其身份签发加密令牌（token）。所有后续请求均需携带该 token，以供服务器验证身份与权限。token 中的身份标识字段由服务器签名，无法被篡改，从而确保认证的安全性。

6.1.1.2授权
平台通过 基于角色的访问控制（RBAC） 模型实现权限管理：
●学者默认拥有学术文献浏览等功能；
●管理员用户拥有系统维护、日志查看、用户管理的全部权限。
授权过程遵循 最小权限原则，确保用户只能访问与其角色相匹配的资源，从而有效防止越权操作。

6.1.1.3输入和数据验证
为防止恶意输入或数据攻击，系统对所有外部输入进行多层验证：
●输入过滤：对用户输入字段进行正则匹配与长度限制，防止SQL注入与XSS；
●文件上传验证：仅允许安全类型文件上传，并对文件内容进行完整性与格式校验；
●敏感数据保护：密码、个人信息等敏感字段在存储前均采用哈希或加密算法处理（如RSA或SHA-256）；
●请求验证：后端对请求头与参数进行验证，防止伪造请求与重放攻击。

6.1.2安全防护框架
6.1.2.1DDoS防护
为防止外界的恶意攻击，如DDoS攻击，本平台使用DDoS流量清洗等方式来有效防护。DDoS流量清洗技术可以实时检测并筛选出恶意流量，将合法流量与攻击流量进行分离，确保平台的正常访问不受影响。
 
6.1.2.2Web防火墙
为防止外界的SQL注入，恶意爬虫扫描等安全问题，本平台使用搭建Web防火墙技术来防止恶意攻击。通过构建Web防火墙，对HTTP请求进行检测，阻止非法的SQL注入，脚本上传，木马上传，从而实现对平台的保护。
 
6.1.2.3数据加密
本平台还使用对敏感信息加密的安全模式，阻止外界恶意窃取敏感信息，信息泄露等问题。通过RSA等方式对数据库信息，计算类信息等进行加密，可以有效防止信息泄露等问题。

6.2异常处理
系统运行过程中不可避免地会出现各种异常情况，包括运行时错误、网络中断、数据库异常或系统组件故障等。为确保系统的高可用性与稳定性，平台设计了完善的 异常检测与异常恢复机制。
6.2.1异常检测
1）当服务器实例在运行过程中遇到未知异常（如空指针、超时、内存泄漏等）时，会立即将异常信息发送至报警系统，触发告警机制。
2）使用一个守护进程，定时向系统各个部分发送存活性检测的消息。若一定时间内未收到某个部分回复此消息就认为此部分故障，并发送报警信息给维护人员。
3）异常检测应根据异常类型进行分类。对于已知的异常类型可以通过自定义的错误码和处理函数进行捕获，而对于未知的异常则通过全局异常捕获机制进行记录和报警，区分异常的严重性，并采取不同的应急处理方案。
 
6.2.2异常恢复
1）事务保障与数据回滚:数据库使用事务进行操作，若发生操作异常或系统宕机，事务会自动回滚，确保数据一致性与完整性。
2）系统故障自恢复:宕机后对现场进行保存。服务器再次重启时确保服务始终可用。
3）备份与恢复策略：制定详细的备份与恢复策略，包括数据库备份、日志备份和文件备份等。备份数据应存储在异地或云存储中，确保灾难发生时能够快速恢复。

6.3缓存
1）服务器前端数据——在客户端浏览器端使用 LocalStorage 缓存用户信息、页面状态等，减少重复请求，提升访问体验。
2）系统采用 MySQL 作为主数据库，并结合 Redis 缓存框架 存储频繁访问的查询结果，减少数据库压力，提高数据读取效率。
3）对动态网页生成的页面内容进行缓存，避免重复渲染，显著提升响应速度。
4）对静态资源（如图片、脚本、样式文件）使用 CDN（Content Delivery Network） 进行分发与缓存。CDN 可根据用户地理位置智能选择最近节点进行内容分发，从而降低延迟、提升系统响应性能与稳定性。

6.4日志管理
日志管理是系统安全与运维的重要组成部分。本平台采用统一的日志收集与分析机制，对运行状态、错误信息与用户操作进行全程记录与保护。
1）服务器实例采用logrus日志框架进行记录，配置定义输出、格式以及日志级别等。
2）采用logrus中可扩展的Hook机制，通过hook定义日志内容，允许使用hook的方式将日志分发到任意地方。
3）系统定期对日志进行清理与归档，过期日志将按配置规则打包并存储至远程备份系统，以便后续审计与问题分析。
4）为防止日志数据被篡改或泄露，平台对日志文件实施严格的访问控制，仅限管理员读取。同时，对涉及用户隐私或敏感信息（如账号、密码、邮箱等）进行脱敏处理，确保数据合规与安全
